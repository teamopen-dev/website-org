Project Lifeguard open for ideas

By Robin van Boven (@Beanow)

September 16, 2019

Photo by Antônia Felipe.

You may have heard, this, before. Many maintainers working on OSS people depend upon are under a lot of stress, responding to people’s wants and needs, while doing their best not to set the world on fire with breaking changes or security vulnerabilities.

It’s impressive how much great software is built this way, but in the same breath, people have been pushing on to the point where having one or two episodes of burnout became normal.

But what options are there for maintainers? They may have realized it’s causing them grief, but posting an abandonment notice and hoping for the best…? That seems like a last resort and something you could feel quite guilty about doing.

Project Lifeguard

What I feel is missing here, is a responsible way to leave your maintainer position. Giving drowning maintainers an escape.

Photo by Jametlene Reskp.

The working idea I have now, is to be a group you can hand off maintenance to which will offer:

  1. Emergency bug and security fixes, by developers we find and pay?
  2. A standard message asking for new maintainers and sponsors.
  3. A long-term plan, to deprecate and archive if needed.

For sponsorships, in particular I think it would be good to look for businesses that are using the software and derive value from it.

The standard message, could be used to make risk a lot clearer for people using the software. Similar to an npm audit. Or the sustainability information proposed here. I’m imagining an output such as:

$ npm install
...
4 packages have been abandoned.
12 packages are receiving emergency maintenance.
Learn what you can do: https://lifeguard.teamopen.dev
Show the package list with: npx lifeguard

Just an idea

For now this is an idea. If you’re interested in helping, get in touch and share your thoughts.
